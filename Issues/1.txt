### 1. State Leakage: `transfer_state.py` Class Attribute Trap

**Status: VULNERABLE**
While you successfully isolated the modern context in `task_context.py` using `dataclasses` and `default_factory`, you left behind a massive global state leak in `colab_leecher/utility/transfer_state.py`.

Because you define `sent_file = []` and `successful_downloads = []` directly on the class body without an `__init__`, these are **class-level mutable attributes**. Any legacy fallback code that instances `Transfer()` will share the exact same list across *all concurrent tasks*. Task A will upload files and append to Task B's lists, resulting in mixed Telegram logs and cross-pollinated Google Drive records.

**The Fix (`colab_leecher/utility/transfer_state.py`):**

```python
class Transfer:
    """Holds state related to file transfer progress and results."""
    def __init__(self):
        self.reset()  # Force instance-level initialization

    def reset(self):
        self.down_bytes = 0
        self.up_bytes = 0
        self.total_down_size = 0
        self.sent_file = []
        self.sent_file_names = []
        self.successful_downloads = []

```

### 2. The "Zombie Slot" Trap: Unbounded `finally` Execution

**Status: VULNERABLE**
In `task_manager.py` -> `taskScheduler`, you properly put `await TASK_QUEUE.release_worker_slot(task_id)` inside the `finally` block (Line 344). However, right above it (Lines 329-331), you `await cancelTask(reason_to_report, task_ctx)`.

If Pyrogram hangs during the `cancelTask` API calls (a common issue with Telegram's backend dropping sockets without closing them), `cancelTask` will block *forever*. The execution will never reach line 344, meaning the worker slot is permanently zombified. After `N` hangs, the bot will refuse all new tasks, requiring a hard reboot.

**The Fix (`colab_leecher/utility/task_manager.py` Lines 329-332):**

```python
        # 2. Dispatch reports/cancellation UI updates safely if task was cancelled or failed
        if _task_error.state or task_ctx.is_cancelled:
            try:
                log.warning(f"taskScheduler finally: Task {task_id[:8]} is failed/cancelled. Dispatching reports.")
                # Wrap in wait_for to guarantee it never blocks the slot release
                await asyncio.wait_for(cancelTask(reason_to_report, task_ctx), timeout=15.0)
            except Exception as cancel_dispatch_err:
                log.error(f"taskScheduler finally: Failed to dispatch reports: {cancel_dispatch_err}", exc_info=True)

```

### 3. Throttling Deadlocks: The Lock-Sleep Catastrophe

**Status: CRITICAL DEADLOCK**
In `task_dashboard.py` -> `update_summary_dashboard`, you have built a system-killing deadlock. The author left a comment claiming to release the lock before sleeping on a `FloodWait`, but the code absolutely does not do this.

At Line 198, the `except FloodWait` block invokes `await asyncio.sleep(wait_secs)` **while still inside** `async with TASK_QUEUE._summary_lock:`. If Telegram issues a 60-second FloodWait, the lock is held for 60 seconds. Every single parallel worker trying to update its progress will stall indefinitely waiting for `_summary_lock`, freezing all background I/O.

**The Fix (`colab_leecher/utility/task_dashboard.py` Lines 198-204):**

```python
                except FloodWait as fw:
                    wait_secs = fw.value
                    log.warning(
                        "FloodWait hit while updating summary dashboard — "
                        "suspending UI edits for %ds.", wait_secs
                    )
                    # Mark the suspension time, but DO NOT sleep here. Return immediately.
                    # The suspension guard at the top of the function handles the logic.
                    TASK_QUEUE._ui_suspended_until = time.monotonic() + wait_secs
                    return TASK_QUEUE.summary_msg

```
