Based on my comprehensive audit of the Colab Telegram Leecher codebase, here are the key issues I've identified:
UI Issues Found:
1. Inconsistent Status Message Handling (handler.py:1384-1391)
   - Photo vs text message editing logic is duplicated with fragile hasattr checks
   - _edit_status_message helper exists but not consistently used across all code paths
2. Upload Error Keyboard Callback Handling (handler.py:1268-1306)
   - The err_action callback handler waits indefinitely with user_decision_event
   - Timeout of 1 hour could leave workers blocked if user never responds
   - No cleanup of keyboard buttons after user decision
3. Dashboard Page Navigation Issues (task_dashboard.py:292-314)
   - Navigation buttons don't update the current page in the backend before rendering
   - Race condition possible when set_dashboard_page() and update_summary_dashboard() are called concurrently
4. Progress Bar Inconsistencies (helper.py:1655-1902)
   - In parallel mode, status_bar updates transfer stats but also tries to edit individual messages
   - Some paths may try to edit a deleted/null status_msg
5. Upload Error Keyboard Doesn't Show Task Context (ui_components.py:411-419)
   - get_upload_error_keyboard shows "Delete Files" / "Keep Files" but doesn't include task ID info in the visible text
Task Manager Issues Found:
1. Worker Slot Leaks (task_context.py)
   - acquire_worker_slot / release_worker_slot can become unbalanced on exceptions
   - The try/finally in taskScheduler uses create_background_task for cleanup, which could fail silently
2. Race Condition in Task Queue Removal (task_manager.py:550-572)
   - TASK_QUEUE.remove_task() and TASK_QUEUE.release_worker_slot() are scheduled as background tasks
   - If multiple tasks finish simultaneously, queue operations may interleave unsafely
3. Dashboard Update Storm (task_dashboard.py:453-510)
   - force_update_summary debounce logic uses time.time() instead of time.monotonic()
   - Could cause drift or incorrect debounce timing
4. Transfer State Inconsistency (task_context.py:90-101)
   - get_current_bytes() and get_percentage() assume down_bytes/up_bytes are lists
   - But SmartBytes may not support sum() the same way
5. Missing Error Propagation in Do_Mirror (task_manager.py:1052-1104)
   - _task_error is accessed as task_ctx.task_error instead of task_ctx.error
   - This would cause AttributeError since the correct attribute is task_ctx.error

Here's my complete audit of all UI (Telegram) issues and task manager issues in the codebase:
---
CRITICAL ISSUES
1. Dual Progress Systems Fighting Each Other
Files: helper.py:1655, progress_dispatcher.py:25, task_dashboard.py:59
Three independent systems update the same Telegram status message:
- status_bar() in helper.py — legacy, called by older code paths
- ProgressDispatcher in progress_dispatcher.py — newer, used by aria2/ytdl
- task_dashboard.py — parallel-mode dashboard
All three check TASK_QUEUE.has_task() to decide whether to edit individual status or skip in favor of dashboard, but they use different throttle timers and different progress models (SmartBytes list vs transfer.down_bytes int). This causes:
- Double-edits on the same message causing MessageNotModified or race conditions
- Dashboard reads stale stats because status_bar() writes to task_ctx.transfer.down_bytes = [done_bytes] (list) while ProgressDispatcher writes transfer.down_bytes[-1] = int(current_bytes) (index update)
- In parallel mode, status_bar() still edits individual messages when force_update=True, contradicting the dashboard model
2. cancel_all_tasks_confirm Double-Removes Tasks from Queue
File: __main__.py:3595-3608
for task_ctx in list(all_tasks.values()):
    await cancelTask("User pressed Cancel All.", task_ctx=task_ctx)  # calls taskScheduler finally block which removes from queue
    await TASK_QUEUE.remove_task(task_ctx.task_id)  # tries to remove again
cancelTask → cancel_task → the taskScheduler finally block already calls TASK_QUEUE.remove_task(). The explicit remove_task() at line 3608 is a double-removal — on the second call it's a no-op, but if the first removal hasn't completed (background task), this causes a race.
3. Upload Error Decision Handler Missing Timeout Cleanup
File: handler.py:267-309
When upload fails, the task:
1. Releases its worker slot (line 273)
2. Waits up to 1 hour for user decision (line 291)
3. Re-acquires worker slot (line 298)
If the user never responds (timeout), keep_files_decision is set to False, but the worker slot is never released again — it was re-acquired at line 298 but the exception at line 309 will propagate up, and the taskScheduler finally block will try to release it. This is fragile and can cause negative worker counts if the finally block doesn't fire.

HIGH-PRIORITY UI ISSUES
4. _edit_status_message Assumes msg.photo Always Has Attribute
File: helper.py:30
async def _edit_status_message(msg, text, reply_markup, parse_mode):
    if msg.photo:
If msg is None or a different type (e.g., after a failed edit recreated it as text), this crashes with AttributeError. The callers at lines 771-776 (handler.py) don't null-check status_msg before calling.
5. Dashboard Photo/Text Mode Mismatch Causes Delete+Recreate Spam
File: task_dashboard.py:349-357
is_photo_msg = bool(hasattr(TASK_QUEUE.summary_msg, 'photo') and TASK_QUEUE.summary_msg.photo)
if is_photo_msg != use_photo:
    await TASK_QUEUE.summary_msg.delete()
When len(summary_text) oscillates around 1024 chars (the photo caption limit), the dashboard repeatedly deletes and recreates the message. Each cycle sends a new message to the owner, flooding the chat.
6. Dashboard move_to_bottom=True Creates Duplicate Messages
File: task_dashboard.py:340-346
The force_update_summary() default is move_to_bottom=True. Each call deletes the old message and sends a new one. During bulk cancellations, multiple force_update_summary calls can interleave, creating multiple summary messages because the delete-and-send isn't atomic.
7. Inconsistent Cancel Button Text Across UI
Files: helper.py:1573, keyboard_layouts.py:59-63, ui_components.py:411-419
Three different cancel button implementations:
- helper.keyboard() → "Cancel" (bare text, no emoji)
- CommonKeyboards.cancel() → "❌ Cancel" (with emoji)  
- MessageTemplate.get_upload_error_keyboard() → "🗑️ Delete Files" / "💾 Keep Files"
Users see inconsistent button styles depending on which code path triggered the message.
8. Status Bar parse_any_size Silently Returns 0 for Valid Input
File: helper.py:1712-1755
The parse_any_size() function in the parallel-mode path of status_bar() uses a regex that expects \d+\.?\d*\s*[a-zA-Z]+ format. But callers pass values like:
- done="3.20 GB" (works)
- done="3.20GB" (no space — regex expects \s* between number and unit, this actually works)
- done="3.20" (no unit — returns 0 because regex fails and fallback only handles pure digits)
This means download progress resets to 0% on certain formatting conventions.

TASK MANAGER ISSUES
9. Background Cleanup Tasks Silently Swallow Errors
File: task_manager.py:552-572
async def _async_cleanup():
    try:
        await TASK_QUEUE.release_worker_slot(task_id)
    except Exception as slot_err:
        log.error(...)  # logged but continues
    
    try:
        await TASK_QUEUE.remove_task(task_id)
    except Exception as remove_err:
        log.error(...)  # logged but continues
If slot release fails, the worker count goes negative. If task removal fails, the task stays in the queue forever, preventing new tasks from starting (if at max capacity). These errors need to be retried or escalated.
10. Do_Mirror Uses task_ctx.task_error (Alias) Instead of task_ctx.error
File: task_manager.py:1076
While this works because task_error is an alias property, it's inconsistent with every other function in the codebase:
- handler.py:44 → _task_error = task_ctx.error  
- handler.py:349 → _task_error = task_ctx.error
- task_manager.py:136 → _task_error = task_ctx.error (in taskScheduler)
Same inconsistency in converters.py (8 occurrences), aria2.py (1), mega.py (1), gdrive.py (4). All use .task_error while the canonical attribute is .error.
11. TaskTransfer.get_speed() Returns Cumulative Average, Not Instantaneous
File: task_context.py:116-121
def get_speed(self) -> float:
    elapsed = time.time() - self.start_time
    current_bytes = self.get_current_bytes()
    return current_bytes / elapsed
This returns average speed since task start, not current speed. After a slow start followed by fast download, the dashboard shows a misleadingly low speed. The last_speed / last_speed_bytes fields (set by status_bar) would be better, but the dashboard calls get_speed() instead.
12. _build_task_paths Missing temp_unzip Path Key
File: __main__.py:167-184
'temp_unzip': f"{task_ctx.work_path}/temp_unzip",
'temp_unzip_path': f"{task_ctx.work_path}/temp_unzip",
Both temp_unzip and temp_unzip_path point to the same location, but the old Paths class uses temp_unzip_path. The temp_unzip key is unused — potential source of confusion if any legacy code references it.
13. status_bar Parallel Mode Writes to Wrong Bytes Field
File: helper.py:1762-1765
if is_upload:
    task_ctx.transfer.up_bytes = [done_bytes]  # REPLACES entire list
else:
    task_ctx.transfer.down_bytes = [done_bytes]  # REPLACES entire list
But ProgressDispatcher (line 73-87) does:
transfer.up_bytes[-1] = int(current_bytes)  # UPDATES last element
If both code paths run for the same task (possible during download→upload transition), one replaces the list [done_bytes] and the other tries to index [-1] — this works but causes inconsistent intermediate states for the dashboard.
14. force_update_summary Debounce Uses time.time() Not time.monotonic()
File: task_dashboard.py:467
now = time.time()
time_since_last = now - TASK_QUEUE._last_force_time
time.time() can jump backward (NTP sync, DST) or forward (leap second). This can cause the debounce to:
- Fire immediately after a clock backward jump (bypassing throttle)
- Skip updates for an extended period after a clock forward jump
15. No Guard Against Re-entrant Dashboard Updates
File: task_dashboard.py:85
async with TASK_QUEUE._summary_lock:
The lock prevents concurrent execution but not re-entrant calls. If update_summary_dashboard is called recursively (e.g., an error handler inside the edit calls force_update_summary), it will deadlock on asyncio.Lock since it's not re-entrant.
