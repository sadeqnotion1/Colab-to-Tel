This is a classic problem when building asynchronous download/upload managers. When you issue a hard `async_task.cancel()`, Python's `asyncio` immediately throws a `CancelledError` at the current `await` point. If Pyrogram is actively chunking and streaming a large file to Telegram's servers, the TCP socket is forcefully torn down without sending the proper MTProto RPC cancellation signals. This results in orphaned streams and ghost files on Telegram's side.

To fix this, we implement a **Cooperative Cancellation Pattern**. We will use the existing `is_cancelled` boolean flag inside `TaskContext` to signal the progress callback. When the progress callback detects this flag, it raises a controlled exception, allowing Pyrogram to cleanly close the socket and abort the RPC request.

Here are the required architectural changes.

### 1. `task_context.py` (Enhancing the Cancellation State)

Your `TaskContext` already has an `is_cancelled` boolean and a `mark_cancelled()` method. We will rely on this flag since it's already integrated. (Using a boolean for cooperative polling in an event loop is perfectly safe and avoids `asyncio.Event` loop-attachment errors during initialization).

Ensure your `mark_cancelled` method looks like this:

```python
    def mark_cancelled(self):
        """Mark task as cancelled and signal gracefully"""
        self.completed_at = datetime.now()
        self.is_cancelled = True

```

### 2. `telegram.py` (Graceful Stream Abort)

We need to modify the `up_progress` callback to actively poll `task_ctx.is_cancelled`. If it's True, we raise an `asyncio.CancelledError`. We also need to intercept this specific error in the main `while` loop so it doesn't trigger the retry logic.

Update the `up_progress` function and the upload loop:

```python
# Add this import at the top if not present
```

### 3. `handler.py` (Soft-Cancel with Hard-Cancel Fallback)

Currently, `cancelTask` immediately invokes `.cancel()` on the `async_task`. We need to refactor this to first set the `is_cancelled` flag, give the `up_progress` callback a few seconds to trigger and clean up the Pyrogram stream, and *only then* execute the hard `.cancel()` if the task is stubbornly hanging.

Refactor the `cancelTask` function:

```python
async def cancelTask(Reason: str, task_ctx: TaskContext = None):
    # ... [Keep initial setup logic] ...

    # 1. TRIGGER THE COOPERATIVE ABORT FLAG
    task_ctx.mark_cancelled()

    # Avoid cancelling/awaiting the currently executing task to prevent self-deadlock.
    current_asyncio_task = asyncio.current_task()
    if (
        task_ctx.async_task
        and task_ctx.async_task is not current_asyncio_task
        and not task_ctx.async_task.done()
    ):
        log.info(f"Requested graceful cancellation for {task_ctx.get_short_id()}. Waiting up to 5s for clean socket teardown...")
        
        # 2. WAIT FOR THE TASK TO GRACEFULLY SHUT DOWN
        wait_time = 0.0
        while not task_ctx.async_task.done() and wait_time < 5.0:
            await asyncio.sleep(0.5)
            wait_time += 0.5
            
        # 3. FALLBACK TO HARD CANCELLATION IF STILL RUNNING
        if not task_ctx.async_task.done():
            log.warning(f"Task {task_ctx.get_short_id()} did not cooperatively exit in time. Forcing hard cancellation.")
            task_ctx.async_task.cancel()
            try:
                await task_ctx.async_task
            except asyncio.CancelledError:
                log.info(f"Task {task_ctx.get_short_id()} successfully hard-cancelled")
            except Exception as e:
                log.error(f"Task {task_ctx.get_short_id()} raised exception during hard cancellation: {e}")
        else:
            log.info(f"Task {task_ctx.get_short_id()} shut down gracefully.")

    # ... [Keep all remaining report generation and cleanup logic as it is] ...

```

### Why this architecture works:

1. **Zero Ghost Files:** Raising `CancelledError` inside the progress callback utilizes Pyrogram's native MTProto stream-handling, allowing it to notify the Telegram API that the file part transmission is intentionally aborted. Telegram immediately discards the partial upload.
2. **Deadlock Prevention:** The 5-second `while not task.done()` watcher ensures we don't hold up the event loop forever if Pyrogram is frozen on a network read.
3. **Safe Cleanup:** Because the abort happens inside standard Python asynchronous execution flow rather than a hard interrupt, it safely unwinds the stack, allowing Pyrogram's internal session context managers to close TCP sockets securely.
